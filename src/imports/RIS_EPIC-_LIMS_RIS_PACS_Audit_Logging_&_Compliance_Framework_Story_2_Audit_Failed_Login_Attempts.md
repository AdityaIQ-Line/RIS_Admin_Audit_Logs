### Primary Actor

- Facility Admin
    
- IT / System Admin
    
- Compliance / Security Officer
    

### Secondary Actors

- Helpdesk / Support Team
    

## User Story Statement

As a **Facility Admin / Security Officer**,  
I want to **track and audit all failed login attempts in RIS**,  
So that I can **detect unauthorized access, prevent brute-force attacks, enforce security policies, and meet regulatory audit requirements**.

## Business Value

- Early detection of security threats
    
- Prevents account compromise
    
- Enables forensic investigation
    
- Supports NABH / ISO 27001 / healthcare security audits
    

## Scope

This feature captures and displays **read-only audit logs** for **all unsuccessful authentication attempts** in RIS.

Included:

- Invalid credential attempts
    
- Locked account login attempts
    
- Expired password attempts
    
- OTP / SSO failures (if applicable)
    

Excluded:

- Successful login events (covered in Login/Logout Audit)
    

## High-Level Workflow

1. User enters login credentials
    
2. Authentication fails
    
3. System records failed attempt
    
4. Retry count updated
    
5. Account locked if threshold exceeded
    
6. Admin reviews failed login analytics & logs
    

## Audit Events Captured

- Invalid Username
    
- Invalid Password
    
- Invalid OTP
    
- Expired Password
    
- Locked Account Attempt
    
- Concurrent Session Restriction Failure (if enabled)

## Validations

### Access Control

- Only **Admin / Security / Compliance roles** can view failed login audits
    
- Facility Admin restricted to **own facility**
    
- Super Admin can view cross-facility data
    

### Data Integrity

- Audit records are **immutable**
    
- No edit / delete / override allowed
    
- Timestamp always generated by server
    

### Attempt Logic

- Every failed authentication attempt must generate exactly **one audit record**
    
- Attempt sequence resets after successful login
    

### Filtering Rules

- From Date ≤ To Date
    
- Mandatory date range if records exceed configured threshold
    
- Username search supports partial match
    

## Edge Cases & Exception Handling

### Authentication Scenarios

- Non-existent username → logged with User ID = null
    
- Multiple rapid attempts → all attempts logged individually
    
- Correct username + wrong password → counted toward lock threshold
    
- Correct credentials after lock → blocked + logged
    

### Account Lock

- Lock triggered after configurable failures
    
- Lock event logged separately
    
- Attempts after lock continue to be logged
    

### System & Network

- Network timeout during auth → logged as failure (timeout)
    

### Data Accuracy

- Timezone normalized to facility timezone
    
- IP spoofing not inferred—raw IP logged only
    

## Non-Functional Requirements

- Logging must not increase login response time
    
- Handles high-volume attack scenarios without data loss
    
- Audit search response ≤ 3 seconds
    
- Retention configurable (e.g., 5–7 years)
    

## Acceptance Criteria

- Every failed login attempt is captured
    
- Failure reason is clearly visible
    
- Account lock events are traceable
    
- Logs are read-only and tamper-proof
    
- Admin can filter by date, user, and IP
    

## Compliance & Security

- Meets healthcare security audit requirements
    
- Supports incident investigation & reporting
    
- Aligns with ISO 27001 access monitoring controls
